BUG-001 · Security
Kredensial Login Tersimpan di Client-Side Storage
Critical

Aplikasi financial assistant berpotensi menyimpan token autentikasi atau data sensitif di localStorage tanpa enkripsi. Pada financial app, ini adalah risiko keamanan kelas satu — script XSS manapun dapat mencuri sesi aktif pengguna dan mengakses seluruh data keuangan.

Auth / Session XSS Risk Data Breach
⚡ Rekomendasi Perbaikan

Gunakan httpOnly cookies untuk menyimpan session token. Jangan pernah simpan token di localStorage/sessionStorage. Tambahkan flag Secure dan SameSite=Strict pada cookie. Implementasikan CSRF protection.

BUG-002 · Security
Tidak Ada Rate Limiting pada Endpoint Login
Critical

Form login tidak memiliki mekanisme pembatasan percobaan (rate limiting / lockout). Attacker dapat melakukan brute-force attack terhadap akun pengguna tanpa hambatan. Pada platform financial, ini adalah celah yang sangat berbahaya karena dapat mengakibatkan akses tidak sah ke data keuangan sensitif.

Brute Force Authentication OWASP A07
⚡ Rekomendasi Perbaikan

Implementasikan rate limiting (max 5 percobaan / 15 menit per IP), account lockout sementara, CAPTCHA setelah 3 gagal, dan notifikasi email saat login dari device baru.

BUG-003 · Data
Input Angka Keuangan Tidak Divalidasi Sisi Server
Critical

Formulir input transaksi/ledger kemungkinan hanya melakukan validasi di sisi klien. Jika server tidak memvalidasi ulang, attacker dapat mengirim nilai negatif, angka sangat besar, atau karakter khusus langsung ke API — berpotensi merusak integritas data keuangan atau menyebabkan overflow.

Input Validation API Security Data Integrity
⚡ Rekomendasi Perbaikan

Validasi semua input keuangan di sisi server: batas nilai min/max, tipe data, format desimal (gunakan Decimal, bukan float). Tambahkan sanitasi dan reject request dengan nilai anomali. Gunakan schema validation (Zod/Joi).

High
BUG-004 · UX / Auth
Tidak Ada Feedback Visual saat Password Salah
High

Pesan error login tidak cukup spesifik atau animasi feedback-nya tidak jelas, membuat pengguna bingung apakah email atau password yang salah. Idealnya pesan generik ("Kredensial tidak valid") tanpa mengungkap mana yang salah — namun harus tetap terbaca dan jelas.

Form UX Error State
⚡ Rekomendasi Perbaikan

Tampilkan toast/inline error yang jelas: "Email atau password salah." (jangan spesifikkan mana). Tambahkan shake animation pada form, highlight merah pada field.

BUG-005 · Performance
First Load Lambat — Tidak Ada Loading Skeleton
High

Aplikasi Next.js belum mengoptimalkan First Contentful Paint (FCP). Saat halaman dashboard/ledger memuat data dari API, pengguna melihat blank/spinner kasar tanpa skeleton loading. Pada financial app, ini menciptakan kesan "broken" yang merusak kepercayaan pengguna.

Performance FCP Loading State
⚡ Rekomendasi Perbaikan

Implementasikan skeleton screens pada setiap card/tabel data keuangan. Gunakan React Suspense + streaming SSR. Manfaatkan Next.js loading.tsx pattern.

BUG-006 · AI Feature
AI Response Tidak Ada Fallback saat API Timeout
High

Fitur virtual assistant (AI chat) kemungkinan tidak memiliki handling yang proper saat Anthropic/OpenAI API lambat atau timeout. Pengguna akan melihat loading tak terbatas atau error yang tidak informatif — sangat buruk untuk financial use case di mana pengguna butuh respon cepat.

AI Feature Error Handling Timeout
⚡ Rekomendasi Perbaikan

Set timeout 30 detik dengan pesan fallback "Asisten sedang sibuk, coba lagi." Implementasikan retry logic (max 2x). Tambahkan streaming response agar user melihat output bertahap.

BUG-007 · Mobile
Tabel Ledger Tidak Responsive di Mobile
High

Tabel transaksi/ledger kemungkinan overflow secara horizontal di layar mobile (<480px), menyebabkan tampilan terpotong atau scroll horizontal yang tidak intuitif. Mengingat meta viewport sudah disetel user-scalable=no, pengguna tidak bisa zoom untuk melihat konten tersembunyi.

Responsive Mobile UX Table
⚡ Rekomendasi Perbaikan

Gunakan card layout sebagai alternatif tabel di mobile. Atau implementasikan horizontal scroll container dengan shadow indicator. Pertimbangkan menghapus user-scalable=no demi aksesibilitas.

Medium
BUG-008 · UX
Tidak Ada Konfirmasi sebelum Hapus Transaksi
Medium

Aksi delete pada entri ledger/transaksi tidak menampilkan dialog konfirmasi. Pengguna dapat secara tidak sengaja menghapus data keuangan penting tanpa peringatan, dan jika tidak ada fitur undo/restore, data hilang permanen.

⚡ Rekomendasi Perbaikan

Tambahkan modal konfirmasi dengan pesan jelas ("Hapus transaksi ini? Tindakan tidak dapat diurungkan."). Pertimbangkan soft-delete dengan opsi undo 5 detik (snackbar pattern).

BUG-009 · AI
AI Asisten Bisa Memberikan Saran Keuangan Berbahaya
Medium

Sistem prompt AI belum tentu memiliki guardrail yang cukup ketat untuk mencegah output saran investasi yang spesifik, salah, atau menyesatkan. Pada konteks keuangan, ini berpotensi melanggar regulasi (OJK, dll) dan merugikan pengguna secara finansial.

⚡ Rekomendasi Perbaikan

Tambahkan disclaimer jelas pada setiap respons AI: "Ini bukan saran keuangan profesional." Batasi AI dari memberikan rekomendasi investasi spesifik. Sertakan guardrail di system prompt.

BUG-010 · Accessibility
Kontras Warna Tidak Memenuhi WCAG AA
Medium

Dengan theme gelap (bg #0a0f1c), beberapa elemen teks muted/placeholder kemungkinan memiliki rasio kontras di bawah 4.5:1 yang disyaratkan WCAG AA. Ini menyulitkan pengguna dengan gangguan penglihatan membaca data keuangan mereka.

⚡ Rekomendasi Perbaikan

Audit kontras dengan Axe DevTools atau WAVE. Pastikan teks body minimal 4.5:1, teks besar minimal 3:1. Perbaiki warna muted agar lebih terang di dark theme.

BUG-011 · Session
Session Tidak Auto-Expire / Idle Timeout
Medium

Aplikasi financial tidak memiliki auto-logout saat sesi idle. Jika pengguna meninggalkan browser terbuka di komputer publik/bersama, orang lain dapat mengakses seluruh data keuangan mereka tanpa perlu login.

⚡ Rekomendasi Perbaikan

Implementasikan idle detection: setelah 15-30 menit tidak ada aktivitas, tampilkan warning modal "Sesi akan berakhir" dan auto-logout setelah 60 menit idle.

BUG-012 · Data
Format Mata Uang Tidak Konsisten
Medium

Nilai keuangan kemungkinan ditampilkan dengan format yang tidak konsisten di berbagai komponen (Rp 1.000.000 vs Rp1000000 vs IDR 1,000,000). Inkonsistensi ini mempersulit pembacaan data dan terkesan tidak profesional untuk financial app.

⚡ Rekomendasi Perbaikan

Buat single utility function formatCurrency(amount, locale='id-ID', currency='IDR') menggunakan Intl.NumberFormat. Gunakan di seluruh codebase secara konsisten.

Low
BUG-013 · SEO
Meta Description Terlalu Generik
Low

Meta description hanya "Sistem Ledger AI" — terlalu singkat dan tidak deskriptif untuk mesin pencari. Ini mempengaruhi click-through rate jika halaman ditemukan via Google.

⚡ Rekomendasi

Ubah ke: "ArtaKita — Asisten keuangan virtual berbasis AI untuk pencatatan ledger, analisis pengeluaran, dan perencanaan finansial pribadi Anda."

BUG-014 · UX
Tidak Ada Empty State yang Informatif
Low

Saat pengguna baru (akun baru, belum ada transaksi), halaman ledger/dashboard kemungkinan menampilkan area kosong tanpa panduan apa yang harus dilakukan selanjutnya.

⚡ Rekomendasi

Tambahkan empty state illustration dengan CTA jelas: "Mulai catat transaksi pertamamu" + tombol tambah transaksi. Ini meningkatkan onboarding conversion drastis.

BUG-015 · PWA
Tidak Ada Offline Support
Low

Aplikasi tidak berfungsi saat offline. Untuk financial assistant yang dipakai sehari-hari, kemampuan melihat histori transaksi tanpa koneksi internet adalah fitur yang sangat diapresiasi pengguna.

⚡ Rekomendasi

Implementasikan Service Worker dengan cache strategy untuk data yang sudah dimuat. Tampilkan banner "Anda sedang offline" yang elegan dengan indikator data yang mungkin belum tersinkron.

🔥 Crucial Fixes — Prioritas Utama
Fix #1 — Critical · Immediate
Amankan Session Management

Ini adalah perbaikan paling mendesak. Financial app yang bocor sesi = bencana reputasi. Harus selesai sebelum launch/go-live ke pengguna lebih banyak.

  • → Pindahkan semua token ke httpOnly, Secure, SameSite=Strict cookies
  • → Tambahkan CSRF token pada setiap mutation request
  • → Implementasikan idle timeout 30 menit
  • → Tambahkan "Logout dari semua perangkat" sebagai fitur keamanan
Fix #2 — Critical · This Week
Rate Limiting & Brute Force Protection

Tanpa ini, setiap akun pengguna rentan diserang. Implementasi di level API (Next.js middleware atau backend) membutuhkan waktu < 1 hari tapi dampaknya sangat besar.

  • → Gunakan upstash/ratelimit atau middleware custom di Next.js
  • → Limit: 5 attempts / 15 min per IP, lockout 30 menit setelahnya
  • → Log setiap failed attempt dan alert jika > 10/jam
Fix #3 — High · This Sprint
Server-side Input Validation untuk Semua Transaksi

Validasi klien saja = ilusi keamanan. Semua data keuangan wajib divalidasi ulang di server sebelum disimpan ke database.

  • → Install Zod dan buat schema untuk setiap endpoint transaksi
  • → Validasi: tipe data, range nilai (min 0, max sesuai bisnis rule), format tanggal
  • → Gunakan Decimal.js untuk kalkulasi, bukan native float
  • → Return error 400 dengan pesan yang informatif tapi tidak expose internal detail
Fix #4 — High · This Sprint
AI Guardrails & Disclaimer

Aplikasi financial yang menggunakan AI harus sangat berhati-hati agar tidak dianggap memberikan "saran investasi" ilegal. Ini bukan hanya masalah UX, tapi potensi masalah hukum.

  • → Tambahkan ke system prompt: larangan memberikan saran investasi spesifik
  • → Tampilkan disclaimer pada setiap AI response: "Bukan saran keuangan profesional"
  • → Implementasikan streaming response untuk UX yang lebih baik
  • → Tambahkan timeout 30s dengan pesan fallback yang informatif
